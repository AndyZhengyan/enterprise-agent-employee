# 基于 OpenClaw PiAgent 的企业数字员工分层演进实施方案（含管理 Portal）

> 说明：本方案以 **L1 基线层** 作为统一工程起点，后续 L2/L3/L4 在同一底座持续叠加；不再采用“独立 MVP 产品线”的建设方式。

## 0. 演进总则（先立规矩，再开工）

1. **一套底座，分层发布**：任何里程碑都必须能直接继承到下一层，不允许“重写式升级”。
2. **先定边界，再加能力**：优先稳定 `Gateway / Runtime / Connector / Governance` 四大边界，再在边界内扩容。
3. **先可观测，再自动化**：没有审计与指标可见性之前，不扩高风险自动化能力。
4. **先灰度，再全量**：模型、技能、连接器升级统一走灰度策略，默认可回滚。

## 0.1 分层能力矩阵（L1→L4）

| 维度 | L1 基线可用 | L2 企业可部署 | L3 企业规模化 | L4 企业完善 |
|---|---|---|---|---|
| Runtime | 单岗位模板 + 单Agent主流程 | 多岗位模板 + 策略路由 | 多Agent协同编排 | 自适应编排与策略优化 |
| Connector | 1 个核心连接器 | 3 个标准连接器 + 准入规范 | 跨域连接器编排 | 统一连接器市场与治理评分 |
| Skill | 白名单加载 | 审批+灰度+回滚 | 技能画像与自动推荐 | 技能资产化运营（ROI归因） |
| Knowledge | 基础RAG | 案例沉淀与版本化 | 知识质量评分与过期治理 | COE驱动的知识生产体系 |
| Governance | RBAC + 人工复核 | 审计追踪 + 预算控制 | 多租户隔离 + 合规分级 | 全链路合规自动巡检 |
| Ops | 基础看板 | SLA/成本看板 | 错误预算 + 运营闭环 | 经营级 ROI 看板 |

## 0.2 历史错误架构清理（本轮已统一）

- 清理“先做独立 MVP、后续整体迁移”的路径，改为“同仓同域模型持续扩展”。
- 清理层级命名冲突：架构层级使用 `L1-L4`，技能层级使用 `S1-S3`。
- 清理不具备审计与回滚的交付方式：未满足门禁能力不得进入下一层。

## 1. L1 基线目标（8 周内）
先在企业级目标不变的前提下，完成第一层可用能力：
- 以 **PiAgent** 作为唯一 Agent 内核。
- 支持 **1 个岗位模板** + **3 个高频任务类型**。
- 打通 **1 个 IM Channel（飞书或企微）** + **1 个企业系统 Connector（如工单/CRM）**。
- 提供一个最小可用的 **企业数字员工管理 Portal**，可查看每个数字员工运行情况与任务情况。

---

## 2. 分层边界（L1 做什么 / L1 暂不做什么）

### 2.1 In Scope（必须做）
1. **Agent 执行闭环**：接收任务 -> 规划执行 -> 调工具 -> 返回结果 -> 状态记录。
2. **岗位化配置（最小版）**：仅支持 1 个岗位模板（Persona/Policy/Workflow）。
3. **基础治理**：
   - RBAC 最小权限
   - 高风险动作人工确认
   - 关键链路审计日志
4. **Portal 最小看板**：
   - 员工运行状态（在线/忙碌/异常）
   - 任务状态（待处理/进行中/已完成/失败）
   - 成功率、失败率、人工接管率、平均耗时

### 2.2 Out of Scope（本期不做）
- 多组织多租户复杂隔离（先单租户）
- 大规模 Skill 市场运营能力（先基础引入与白名单）
- 复杂多 Agent 协同编排（先单 Agent 为主）
- 高级财务/法务自动审批（默认人工复核）

> 以上能力并非“不要”，而是进入后续 L2/L3/L4 叠加清单。

## 3. L1 功能清单

## 3.1 PiAgent Runtime（L1）
- 任务执行流程（Plan/Act/Reflect）
- 基础失败重试（最多 2 次）
- 人工接管（失败或高风险动作）

**验收**
- 3 类任务可稳定跑通，自动完成率 >= 50%。

## 3.2 Channel + Connector（L1）
- Channel：飞书或企微二选一
- Connector：工单系统或 CRM 二选一
- Skill：检索、总结、更新记录（3 个基础 Skill）

**验收**
- 用户可在 IM 发起任务并收到结构化结果卡片。

## 3.3 Portal（L1，重点）

### A. 员工列表页（Digital Employees）
- 字段：员工名、岗位、当前状态、24h 成功率、运行中任务数、最近异常时间
- 支持按岗位/状态筛选

### B. 员工详情页（Employee Detail）
- 基本信息：岗位模板版本、权限范围、启用 Skill 列表
- 运行指标：今日任务数、成功率、平均耗时、人工接管率
- 最近任务：最近 20 条任务（状态、耗时、失败原因）

### C. 任务中心（Tasks）
- 任务列表与状态筛选
- 任务详情（输入、执行步骤、工具调用记录、结果、错误原因）
- 人工操作（重试、转人工、关闭）

### D. 运行告警（Alerts）
- 告警类型：失败率突增、连接器异常、权限拒绝、模型超预算
- 告警处理状态：待处理/处理中/已恢复

**验收**
- 运营人员可在 Portal 中定位“哪个员工、哪类任务、为什么失败”。

## 5.1 向上兼容设计约束（必须落地到接口与数据）

### A. 数据模型约束
- `task.status` 采用可扩展枚举，新增状态只能追加，不得覆写语义。
- `task_step` 必须保留 `tool_name` 与快照字段，用于后续 L3 根因归因。
- `audit_log.trace_id` 为全链路主键，后续所有新模块必须引用。

### B. API 约束
- 读接口优先向后兼容（字段只增不减）。
- 写接口新增能力用可选参数，不以破坏性变更替代旧行为。
- 所有高风险动作接口需预留 `approval_context` 字段以支持 L2 审批编排。

### C. 部署与配置约束
- 所有关键策略（路由、重试、限流）必须配置化，不允许硬编码。
- 每次发布必须记录变更版本与回滚目标版本。

## 6. L1 实施步骤（8 周）

### Week 1-2：内核与主流程
- PiAgent Runtime 部署
- IM Channel 打通
- 任务主流程打通

**里程碑**：可从 IM 创建任务并返回结果。

### Week 3-4：工具接入与任务闭环
- 接入 1 个企业系统 Connector
- 上线 3 个基础 Skill
- 接入任务状态持久化（task/task_step）

**里程碑**：3 类任务闭环跑通。

### Week 5-6：Portal 最小功能
- 员工列表页 + 详情页
- 任务中心页（列表 + 详情 + 重试）
- 基础指标 API 与图表

**里程碑**：可看见每个员工运行与任务情况。

### Week 7：治理与告警
- RBAC 最小权限
- 审计日志接入
- 告警规则与 Alerts 页面

**里程碑**：异常可发现、可追溯。

### Week 8：联调与试点上线
- 压测与稳定性优化
- 试点部门上线
- 周度复盘机制建立

**里程碑**：满足试点可用标准。

## 7. L1 成功标准（试点）
- 自动完成率 >= 50%
- 人工接管率 <= 40%
- P1 任务平均处理时长下降 >= 30%
- Portal 可在 10 分钟内定位异常任务根因
- 试点用户满意度 >= 4.2 / 5

- 前端工程师 x1：Portal（员工页/任务页/告警页）

## 9. 分层叠加路线（同一底座持续扩展）

### L2 企业可部署层（2-4个月）
- 岗位从 1 扩展到 3-5（销售运营、客服、HR 助理等）
- Connector 从 1 扩展到 3，并建立连接器准入规范
- Skill 管理从白名单升级为“审批 + 灰度 + 回滚”
- 引入双模型路由、限流与成本看板
- 引入配置中心（版本管理 + 灰度发布）

### L3 企业规模化层（4-8个月）
- 多租户与组织隔离（租户级数据、权限、审计边界）
- 多 Agent 协同流程（任务分解、角色协作、冲突解决）
- 完整 Badcase 运营闭环（自动归因 + 修复工单）
- 可观测性升级（SLA、SLO、错误预算）

### L4 企业完善层（8-12个月）
- COE 治理机制常态化（技能评审、知识评审、风险评审）
- 全链路合规（留痕、脱敏、分级留存、删除策略）
- ROI 体系进入业务考核（部门级节省工时与提效）
- 形成可复制的岗位装配工厂（模板化上线新岗位）

### 分层实施原则（关键）
1. **同一域模型持续演进**：`digital_employee / task / task_step / alert / audit_log` 只做向后兼容扩展，不做推倒重建。
2. **同一服务边界持续加厚**：Gateway、Runtime、Connector、Ops 逐层增强，而不是每层重写。
3. **同一交付节奏持续复盘**：每层都以“可观测 + 可审计 + 可回滚”为默认交付标准。

## 10. 反模式清单（避免“看似快，实则返工”）

- 在 L1 里把业务规则写死在 Prompt，不沉淀到策略层。
- 为了演示速度绕过审计日志，导致后续无法做 Badcase 归因。
- 连接器直接耦合业务域对象，导致 L2 扩展时大面积重构。
- 把“临时脚本”当长期能力，不纳入版本、测试与回滚流程。
